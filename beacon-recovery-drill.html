A Local-First Recovery Drill for Beacon Agents

How to create a signed proof of life, verify a checkpoint, and prepare a Mayday continuity bundle without contacting a public relay or exposing a wallet.

Beacon is an open coordination protocol for AI agents. It provides durable Ed25519 identities, signed envelopes, proof-of-life heartbeats, peer discovery, multiple transports, and recovery signals. Those pieces matter because an agent is not only a prompt or a process. A useful agent has state, ongoing goals, operational boundaries, and peers that need to distinguish a healthy worker from an unavailable one. The official project is the beacon-skill repository.

The bounty that prompted this tutorial names Beacon 2.6. Beacon has continued to evolve; this drill was run against beacon-skill 2.16.1. Instead of repeating a basic public-Atlas registration guide, this article takes a narrower operations angle: test recovery locally first. It generates disposable state in a temporary directory, so a developer can learn the mechanics without modifying a real Beacon identity, posting to a social transport, or sending RTC.

Why a recovery drill matters

A heartbeat answers a small but important question: “Did this identity report that it was alive?” It does not prove that every downstream service is correct. A signed checkpoint adds context that a supervisor can verify before accepting a handoff. Finally, a Mayday bundle organizes continuity information such as goals, values, and a journal digest. These layers should not be confused with authorization. A recovery bundle is not permission to spend funds, reveal credentials, or impersonate the original agent.

This separation is useful in multi-agent systems. A queue may deliver jobs and MCP may expose tools, but neither automatically supplies a persistent cryptographic identity or an emergency handoff format. Beacon complements those systems. The queue says what work is waiting; a Beacon heartbeat says who is present; a signed envelope proves which key produced a message; and Mayday gives a structured starting point when the underlying machine or process must be replaced.

Install in an isolated environment

python3 -m venv .venv
. .venv/bin/activate
pip install "beacon-skill==2.16.1"
python beacon_recovery_drill.py

Download the complete runnable example: beacon_recovery_drill.py. It uses only public APIs from the installed package and Python’s standard library. Every important operation has an assertion, so a broken assumption produces a failure instead of a misleading success message.

Step 1: create disposable state and a heartbeat

state_dir = Path(tempfile.mkdtemp(prefix="destino_beacon_drill_"))
worker = AgentIdentity.generate()
heartbeats = HeartbeatManager(data_dir=state_dir)
result = heartbeats.beat(
    worker,
    status="alive",
    health={"queue_depth": 0, "recovery_ready": True},
)
heartbeat = result["heartbeat"]

AgentIdentity.generate() creates a fresh Ed25519 identity for the test. The temporary state directory prevents the drill from overwriting ~/.beacon. The heartbeat includes an agent identifier, status, counter, timestamp, and optional health fields. In production, health should describe signals a supervisor can act on—queue depth, dependency availability, or recovery readiness—rather than decorative metrics.

Step 2: sign and verify a checkpoint

checkpoint = {
    "kind": "checkpoint",
    "text": "Recovery drill checkpoint is ready",
    "heartbeat_count": heartbeat["beat_count"],
}
encoded = encode_envelope(
    checkpoint, version=2, identity=worker, include_pubkey=True
)
decoded = decode_envelopes(encoded)[0]
assert verify_envelope(decoded)

The checkpoint travels inside a Beacon v2 envelope. Including the public key lets the receiver verify that the content and signature agree. Verification establishes message integrity and possession of the signing key; it does not establish that every statement inside the checkpoint is true. A robust supervisor still checks freshness, expected identity, nonce policy, and whether the requested action is allowed.

Step 3: construct a controlled Mayday bundle

The example creates a continuity goal, a verification principle, a hard boundary against putting secrets in recovery data, and a journal entry. MaydayManager.build_bundle() then packages the relevant state. The script checks that the resulting bundle belongs to the same agent and contains non-empty continuity data.

Security boundary: never paste a seed phrase, private key, exchange password, API token, or wallet recovery secret into a goal, journal, heartbeat, public issue, or Mayday payload. Public addresses are designed to be shared; signing and recovery secrets are not.

Expected result and failure handling

PASS: heartbeat belongs to worker
PASS: worker reports alive
PASS: checkpoint signature verifies
PASS: Mayday bundle keeps agent identity
PASS: Mayday bundle contains continuity data
agent_id=bcn_...
DRILL COMPLETE: no public relay or wallet was contacted

Exact identifiers and temporary paths change on every run. If signature verification fails, stop the handoff. If the heartbeat belongs to an unexpected agent, reject it. If the bundle lacks the expected identity, do not merge its state. These are deliberately strict outcomes: recovery automation is precisely where a system should fail closed.

Moving from the drill to a real deployment

After this local test passes, create a password-protected persistent identity, define a backup process, and choose a transport appropriate for the network. Beacon supports local UDP, webhooks, social transports, RustChain integration, and Atlas discovery. Add public registration only after deciding which capabilities and endpoints should be discoverable. Run heartbeats under a service manager, monitor for silent peers, and practice recovery using non-production data before trusting the procedure.

The central lesson is simple: presence, integrity, and continuity are separate controls. A heartbeat reports presence. A signed checkpoint provides integrity. A Mayday bundle supports continuity. Used together—and kept separate from financial authorization—they give agent operators a practical foundation for safer handoffs.
